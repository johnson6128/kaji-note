# エラーハンドリング設計 — kaji-note

本書はアプリ全体のエラー処理方針を定義する。  
個別機能の仕様（edge-functions-api.md の共通エラーコード等）はここに準拠する。

---

## 基本原則

1. **ユーザーに技術的詳細を見せない** — エラーコードや HTTP ステータスは内部ログにのみ記録し、画面には日本語の説明文を表示する
2. **操作継続を最優先** — 一機能のエラーが他機能を止めない。AI が使えなくても手動入力は続けられる
3. **回復可能性を明示** — 「しばらく待てば直る」「端末の操作が必要」「今日は使えない」を区別してユーザーに伝える
4. **自動リトライは控えめに** — ユーザーの知らない裏での無限リトライは課金増・バッテリー消費に繋がる。回数を上限で絞る
5. **オフライン状態は「エラー」ではなく「制限状態」** — 閲覧は正常動作、書き込み系のみ非活性として案内する

---

## エラー分類

| レベル | 定義 | 通知手段 | 例 |
|--------|------|----------|-----|
| **Critical** | アプリが使用不能になる | モーダル | 認証切れでセッション無効 |
| **Error** | 操作が失敗し、ユーザーアクションが必要 | インラインメッセージ + 再試行ボタン | 写真アップロード失敗、保存失敗 |
| **Warning** | 機能が制限されるが操作継続可能 | トースト / バナー | AI枠超過、オフライン状態 |
| **Info** | 結果の通知 | トースト（自動消去） | 保存完了、実施記録完了 |

---

## UX パターン（通知手段の定義）

### トースト（3秒自動消去）

- 用途: Warning / Info レベル。操作完了通知、一時的な機能不能の案内
- 画面下部に表示。複数同時表示は最大 1 件（後発が上書き）
- 再試行ボタンは持たない

### インラインエラー

- 用途: Error レベル。特定のステップ・フォーム項目に紐づくエラー
- 対象要素の直下に赤文字で表示
- 写真アップロード失敗時はサムネイル枠内に「↺ 再試行」ボタンを表示

### オフラインバナー

- 用途: オフライン継続中に画面上部に固定表示
- 「オフライン — 閲覧のみ利用できます」
- オンライン復帰時に自動消去し、バックグラウンド同期を開始する

### モーダルダイアログ

- 用途: Critical エラー、または破壊的操作の確認
- 本文に回復手順を含める（例:「再ログインしてください」）
- キャンセル不可の場合のみ単ボタン構成にする

### スケルトン + 再取得ボタン

- 用途: 一覧・詳細の初回データ取得失敗
- 「読み込みに失敗しました（再試行）」ボタンを中央表示

## シナリオ別ハンドリング

### 写真アップロード失敗

写真アップロードはテキスト保存と**分離**して扱う。  
テキスト情報を先に保存し、写真だけを非同期でアップロードする設計にする。

| 状況 | 挙動 |
|------|------|
| アップロード中のネットワーク切断 | インラインエラー表示。「↺ 再試行」でリトライ。オフラインバナーを表示 |
| タイムアウト（30秒） | 同上。自動リトライ 1 回（オンライン確認後） |
| ファイルサイズが圧縮後も 500KB 超 | アップロード前にバリデーション。「写真を選び直してください」インラインエラー |
| Storage 権限エラー（403） | トースト「写真を保存できませんでした。グループ設定を確認してください」 |
| 連続失敗（2 回） | 自動リトライを停止。インラインエラーに「↺ 後で再試行」ボタンを残す |

**オフラインキュー方針:**  
アップロード待ちの写真は MMKV にローカルパスと対象ステップ ID を記録する。  
オンライン復帰時に TanStack Query の `onSuccess` / `useNetInfo` を組み合わせて自動リトライする。  
ただし、アプリを閉じて再起動した場合は自動リトライしない（データ消失より意図しない通信を避ける）。

---

### ネットワーク切断

`@react-native-community/netinfo` でオンライン状態を監視する。

| 状態変化 | 挙動 |
|----------|------|
| オンライン → オフライン | オフラインバナーを表示。書き込み系 UI を非活性化 |
| オフライン中の書き込み操作タップ | 「オフライン中は保存できません。接続を確認してください」トースト |
| オフライン → オンライン復帰 | バナーを消去。TanStack Query の `invalidateQueries` でデータを再取得 |

### Gemini API 制限超過 / AI エラー

AI 機能はすべて**オプション扱い**。エラーが起きても手動入力で作業を継続できる。

| エラーコード | HTTP | ユーザー向けメッセージ | 挙動 |
|-------------|------|----------------------|------|
| `QUOTA_EXCEEDED` | 429 | 「AI機能の本日の利用上限に達しました。明日以降お試しください」 | AI ボタンをセッション中グレーアウト。リトライしない |
| `UPSTREAM_ERROR` | 502 | 「AI機能が一時的に使えません。手動で入力してください」 | 1 回のみ自動リトライ（3 秒後）。失敗でトースト表示 |
| `INTERNAL_ERROR` | 500 | 同上 | 同上 |
| `UNAUTHORIZED` | 401 | 表示しない（セッション再認証フローへ） | Critical エラー処理へ委譲 |
| オフライン時 | — | ツールチップ「オンライン時のみ利用できます」 | AI ボタンを非活性化（押せない） |

**QUOTA_EXCEEDED の当日無効化:**  
発生時刻を MMKV に保存し、翌日 0:00 以降まで AI ボタンを無効にする。  
アプリ再起動でフラグを確認し、期限切れなら自動解除する。

---

### 認証エラー

| 状況 | 挙動 |
|------|------|
| JWT 自動更新成功 | ユーザーに通知しない（透過的に処理） |
| JWT 更新失敗（セッション切れ） | モーダル「セッションが切れました。再ログインしてください」→ ログイン画面へ遷移 |
| Google OAuth 失敗 | トースト「Googleログインに失敗しました。もう一度お試しください」 |

---

### データ取得失敗

| 画面 | 挙動 |
|------|------|
| 手順書一覧 | スケルトン → エラー時「読み込みに失敗しました（再試行）」ボタン |
| 手順書詳細 | 同上。キャッシュが存在すれば古いデータを表示し「オフラインで表示中」を小文字で付記 |
| 実行モード（オフライン） | キャッシュがあれば表示継続。実施記録はオンライン復帰後に送信する |

---

### 保存失敗（手順書・ステップ）

| 状況 | 挙動 |
|------|------|
| サーバーエラー（5xx） | インラインエラー「保存に失敗しました」＋「再試行」ボタン |
| バリデーションエラー（400） | 問題フィールドのインラインエラー。自動リトライなし |
| 権限エラー（403） | トースト「この操作を行う権限がありません」 |
| オフライン | 書き込み操作はオフライン時非活性のため到達しない（防御的に上記と同扱い）|

## リトライ戦略まとめ

| エラー種別 | 自動リトライ | 上限 | 待機 |
|-----------|------------|------|------|
| ネットワークタイムアウト | する | 2 回 | 2 秒、4 秒 |
| サーバーエラー 5xx | する | 1 回 | 3 秒 |
| AI `UPSTREAM_ERROR` / `INTERNAL_ERROR` | する | 1 回 | 3 秒 |
| AI `QUOTA_EXCEEDED` | **しない** | — | — |
| 4xx クライアントエラー | **しない** | — | — |
| 写真アップロード失敗 | する（手動トリガー後） | 2 回 | 2 秒、4 秒 |

---

## エラーログ方針

- **ユーザー向け**: 日本語の説明文のみ表示し、エラーコード・スタックトレースは非表示
- **内部ログ**: `console.error` にエラーコード・HTTP ステータス・発生箇所を出力（開発・ステージング環境）
- **本番モニタリング**: 初期リリース後に Sentry などを導入するまでは Supabase の Edge Functions ログで代替
- **個人情報**: ログにメールアドレス・写真内容を含めない
