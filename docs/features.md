# 機能定義 — kaji-note

ユースケース定義（usecases.md）が「誰が何をしたいか」を記述するのに対し、  
本書はシステムが「何ができるか・どう振る舞うか」を定義する。

---

## F-01 認証・アカウント

### F-01-1 メール認証

| 項目 | 仕様 |
|------|------|
| メールアドレス | RFC 5322 形式を検証。255 文字以内 |
| パスワード | 8 文字以上。英字・数字をそれぞれ 1 文字以上含む |
| 重複登録 | 同一メールアドレスが既存の場合はエラーメッセージを表示 |
| セッション | Supabase Auth のセッショントークンを端末に保存し、アプリ再起動後も維持する |
| ログアウト | セッショントークンを破棄し、認証画面へ遷移する |

- **初期表示名**: サインアップ直後に `handle_new_user()` トリガーが自動設定する。メール認証の場合はメールアドレスのローカルパート（`@` より前の部分）、Google OAuth の場合は Google アカウントの表示名を使用する。初期値は F-01-3（プロフィール編集）でいつでも変更可能。

### F-01-2 Google ソーシャルログイン

- Supabase Auth の Google OAuth プロバイダーを使用する
- 初回ログイン時は自動でアカウントを作成する
- Google アカウントの表示名・アイコンを初期プロフィールとして取り込む

### F-01-3 プロフィール

| 項目 | 仕様 |
|------|------|
| 表示名 | 1〜30 文字。空文字は不可 |
| アイコン画像 | JPEG / PNG。端末側で 256×256px・100KB 以下に圧縮してからアップロード |

## F-02 グループ管理

### F-02-1 グループのデータ制約

| 項目 | 仕様 |
|------|------|
| グループ名 | 1〜50 文字 |
| アイコン画像 | JPEG / PNG。256×256px・100KB 以下に圧縮 |
| メンバー上限 | 20 名（アプリケーション層で施行。グループ参加処理前に件数確認し超過時はエラー。DB には CHECK 制約なし） |

### F-02-2 権限モデル

| 操作 | 管理者 | 編集メンバー | 閲覧メンバー |
|------|:------:|:-----------:|:-----------:|
| 手順書の閲覧 | ○ | ○ | ○ |
| 手順書の作成・編集 | ○ | ○ | — |
| 手順書の削除 | ○ | 自分が作成したもののみ | — |
| 実施済み記録 | ○ | ○ | ○ |
| メンバー招待 | ○ | — | — |
| メンバーの権限変更・除名 | ○ | — | — |
| グループ名・アイコンの変更 | ○ | — | — |

- グループはユーザー 1 人あたり複数所属可能
- グループ作成者は自動的に管理者になる
- グループ削除は初期リリース対象外（将来機能）

### F-02-3 招待リンク

| 項目 | 仕様 |
|------|------|
| 有効期限 | 発行から 7 日間 |
| 用途 | 1 リンクで複数人が参加可能（使い捨てではない） |
| デフォルト権限 | 招待リンクから参加したメンバーは「閲覧メンバー」として登録される |
| 再発行 | 管理者は既存リンクを無効化して新しいリンクを発行できる |

## F-03 手順書

### F-03-1 データ制約

| 項目 | 仕様 |
|------|------|
| タイトル | 1〜100 文字 |
| カテゴリ | 選択式（掃除 / 料理 / 洗濯 / 収納 / その他） |
| ステップ数 | 最大 30 ステップ |
| ステップテキスト | 1 ステップあたり最大 500 文字 |
| 写真枚数 | 1 ステップあたり最大 3 枚 |
| 公開状態 | 下書き（作成者のみ閲覧）/ 公開（グループ全員が閲覧可）の 2 状態 |

### F-03-2 写真アップロード

- アップロード前に端末側で以下の処理を行う

| 処理 | 仕様 |
|------|------|
| リサイズ | 長辺 1280px 以内にリサイズ |
| 圧縮 | JPEG 品質 80 相当。変換後ファイルサイズ上限 500KB |
| フォーマット | JPEG に統一（PNG・HEIC から変換） |

- Supabase Storage の `step-photos` バケットに保存し、DB にはファイルパスを保存する

### F-03-3 ステップの並び替え

- ドラッグ＆ドロップで順序を変更できる
- 順序は `position` 列（整数）で管理する
- 保存ボタンを押すまで変更は確定しない（サーバーには即時送信しない）

### F-03-4 削除

- **論理削除**を基本とする（`deleted_at` タイムスタンプを付与）
- 削除された手順書は一覧に表示されないが DB 上は保持する
- 物理削除は管理者が明示的に操作した場合のみ（将来機能、初期リリースでは未実装）
- 手順書削除時、紐づく写真ファイルも Storage から削除する

**Storage ファイルの削除タイミング（アプリ層の責務）:**

| 操作 | Storage への影響 |
|------|----------------|
| 手順書の論理削除（`soft_delete_note`） | 写真は Storage に残す。物理削除実装時に一括削除（初期リリースでは未実装） |
| ステップから写真を明示的に削除（編集画面で削除） | アプリがその写真の Storage オブジェクトを即時削除してから保存する |
| ステップごと削除（ステップを丸ごと消す） | そのステップに紐づく全写真を Storage から削除してから保存する |

## F-04 頻度・スケジュール

### F-04-1 頻度の種類

| 種類 | 設定内容 |
|------|---------|
| 設定なし | 次回予定日を計算しない |
| 毎日 | 設定のみ（曜日指定なし） |
| 週次 | 曜日を 1 つ以上選択（複数可） |
| 月次 | 日付を 1 つ指定（例: 15 日）。月末を超える場合は月末日に丸める |
| 季節ごと | 3 ヶ月ごとに自動計算 |
| カスタム | N 日ごと（N は 1〜365 の整数） |

### F-04-2 次回実施予定日の計算

- 「実施済み」にした日を起点として次回予定日を計算する
- 一度も実施していない場合は手順書の `created_at::DATE` を起点とする
- 週次で複数曜日を指定した場合: 実施日以降で最も近い指定曜日を次回予定日とする（例: 月・木指定で水曜に実施 → 次回は木曜）
- 計算結果は `next_scheduled_date` 列に保存し、一覧表示に使用する
- **新規作成時の初期値**: 頻度が「設定なし」以外の場合、クライアントが INSERT 時に現在日付（INSERT 送信時のローカル日付）を起点として計算した値を `next_scheduled_date` に含める（DB が返す `created_at` を待たずに計算する）

### F-04-3 バッジの表示ロジック

バッジは以下の優先順位で評価する（上位条件が一致した時点で適用）:

| 優先順位 | 条件 | バッジ |
|---------|------|--------|
| 1 | 当日に「実施済み」を記録済み | ✅ 実施済 |
| 2 | `next_scheduled_date` ≤ 本日 | 🔴 期限超過 |
| 3 | `next_scheduled_date` > 本日 | 🕐 〇〇日（具体的な日付） |
| 4 | 頻度未設定 | バッジなし |

## F-05 閲覧・実行モード

### F-05-1 手順書一覧

- ソート順: デフォルトは「期限超過 → 近い予定日順 → 予定なし（`created_at` の新しい順）」
- 検索: タイトル・カテゴリの前方一致・部分一致で絞り込む
- 表示対象: 自分が所属するグループに公開されている手順書（自分の下書きを含む）

### F-05-2 実行モード

- 実行中のステップ位置は Zustand で管理する（サーバーに保存しない）
- アプリをバックグラウンドにしても進捗は保持する（フォアグラウンド復帰時に再開可能）
- ステップの閲覧・進捗操作はオフラインでも可能（キャッシュ済み手順書の場合）
- 最終ステップ完了時: `executions` テーブルに実施日時・実行者を記録し、`next_scheduled_date` を更新する。**オフライン時は「完了・実施済みにする」ボタンを非活性化する**（オンライン復帰後に再度実行する必要がある）
- **下書きノートでの実行モード**: 実行モード開始ボタンは下書きノートでも表示する（手順の動作確認が目的）。ただし最終ステップの「完了・実施済みにする」ボタンは下書き状態では非活性化し、「公開してから実施を記録してください」と案内する。`executions` テーブルへの INSERT は公開ノートのみ RLS が許可する

## F-06 共有

### F-06-1 グループ内公開

- 下書き → 公開の切り替えは手順書ごとに行う
- 公開された手順書はグループ全メンバーの一覧に表示される

### F-06-2 閲覧専用リンク

| 項目 | 仕様 |
|------|------|
| 権限 | 閲覧のみ（編集・実施記録の書き込み不可） |
| 有効期限 | 無期限（admin が手動で無効化可能。creator が editor の場合は admin に依頼） |
| 対象 | リンク発行時点で公開状態の手順書 1 件 |
| 認証 | 不要（未ログインのブラウザでも閲覧可能） |

- **共有 URL 形式**: `{EXPO_PUBLIC_SHARE_BASE_URL}/share/{token}`。Expo Router の Web ビルド（S-06）が `get-note-by-share-token` Edge Function を呼んでノートを表示する。URL はアプリの共有ボタンからクリップボードにコピーされる
- **閲覧アクセスの仕組み**: 未認証ユーザーからのアクセスは `get-note-by-share-token` Edge Function を経由する。Edge Function が `service_role` でトークン検証・ノート＋ステップデータ取得を行い、認証不要でレスポンスを返す
- **発行権限**: `can_edit_in_group`（editor / admin）はリンクを作成できる。**無効化**は `is_group_admin` のみ可能
- **手順書が下書きに戻された場合**: `get-note-by-share-token` は `note.status = 'published'` を条件に含めてチェックする。下書き（`draft`）状態の手順書はトークンが有効でもデータを返さず 404 を返す。share_link レコードは残るため、再度公開状態に戻せば同じトークンで閲覧可能になる

## F-07 AI 機能

Gemini API の呼び出しは Supabase Edge Functions 経由で行い、クライアントに API キーを露出しない。

### F-07-1 タイトルからステップ自動生成

| 項目 | 仕様 |
|------|------|
| トリガー | 「✨ AIで生成」ボタンをタップ（自動発火なし） |
| 入力 | タイトル文字列 |
| 出力 | ステップテキストの配列（最大 10 ステップ） |
| 既存ステップの扱い | 生成前に確認ダイアログを表示し、承認時は既存ステップを置き換える |

### F-07-2 次ステップ提案

| 項目 | 仕様 |
|------|------|
| トリガー | ステップ入力欄の「✨」ボタンをタップ |
| 入力 | タイトル＋既存ステップ全文 |
| 出力 | 次ステップ候補のテキスト 1 件。ユーザーが編集・確定する |

### F-07-3 写真からステップ説明文生成

| 項目 | 仕様 |
|------|------|
| トリガー | 写真添付直後に自動で呼び出す（ボタン不要） |
| 入力 | AI用に再圧縮した写真（Base64）＋タイトル |
| 出力 | ステップ説明文案。既存テキストが空の場合のみ自動入力し、入力済みの場合は「AIの提案」として別表示 |

**写真圧縮の 2 段階フロー**（撮影 / ライブラリ選択後の処理順序）:

1. **AI 用圧縮** — 長辺 800px・JPEG 品質 70・base64 後 400KB 以内に圧縮 → `ai-describe-photo` Edge Function へ送信
2. **Storage 用圧縮** — 長辺 1280px・JPEG 品質 80・500KB 以内に圧縮 → Supabase Storage へアップロード

AI に送る画像はトークン消費を抑えるため Storage 用より小さく圧縮する。両処理は並列ではなく順番に実行し（AI 送信を先に行う）、いずれかが失敗してもステップ入力そのものは続行できる。

### F-07-4 音声入力

| 項目 | 仕様 |
|------|------|
| トリガー | 「🎤」ボタンを長押し（押している間録音・離して確定） |
| 第 1 段階 | 端末 OS の音声認識 API でテキスト変換（オフライン認識を優先） |
| 第 2 段階 | 変換テキストを Gemini API に送り「手順文として自然な一文」に整形 |
| 出力 | 整形済みテキストをステップ入力欄に挿入。編集・再録音が可能 |

- **オフライン時の動作**: 第 1 段階（端末 OS STT）はオフラインでも動作する。第 2 段階（Gemini API）はオンライン必須。オフライン時は AI 整形をスキップし、端末認識テキストをそのままステップ入力欄に挿入する

### F-07-5 エラー・レート制限

| 状況 | 挙動 |
|------|------|
| API エラー（5xx） | トースト通知「AI機能が一時的に使えません」を表示。手動入力は継続可能 |
| 無料枠超過（429） | 同上。翌日リセットされる旨を案内 |
| オフライン時 | AI ボタンを非活性化し、ツールチップ「オンライン時のみ利用可能」を表示 |

## F-08 オフライン対応

### F-08-1 キャッシュ対象

| データ | キャッシュ方法 |
|--------|-------------|
| 手順書一覧（テキスト） | TanStack Query + AsyncStorage persister |
| 手順書詳細（テキスト） | 同上。一度開いた手順書を自動キャッシュ |
| ステップ写真 | 一度表示した画像を端末ローカルにキャッシュ（MMKV で URI 管理） |

### F-08-2 オフライン時の制限

以下の操作はオフライン時に**不可**（UI 上で非活性または案内を表示）:

- 手順書の新規作成・編集・保存
- 写真のアップロード
- 実施済みの記録
- AI 機能の利用（例外: 音声認識の第 1 段階〔端末 OS STT〕はオフラインでも使用可）
- メンバー招待・グループ設定の変更

### F-08-3 オンライン復帰時

- TanStack Query がバックグラウンドで最新データをフェッチし、キャッシュを更新する
- キャッシュと最新データの競合は「サーバー優先」で解決する（クライアントの未保存変更は破棄）
